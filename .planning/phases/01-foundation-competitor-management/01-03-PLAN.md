## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → createCompetitor Server Action | Server Functions are reachable via direct POST, not just the UI (Next.js docs); no auth by design |
| User-supplied URL → database → Phase 2 fetcher | SSRF seed: stored URLs get fetched server-side later |
| Dev machine → Neon (drizzle-kit push) | Direct DB credentials used by tooling |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02 | Tampering / Info Disclosure | Stored monitored URLs (SSRF) | mitigate | createCompetitor parses ALL input through createCompetitorSchema (scheme allowlist + internal-host denylist from Plan 02); UI surfaces the block with UI-SPEC copy |
| T-01-05 | Elevation of Privilege | Public unauthenticated Server Action | accept | Explicit project decision (auth out of scope, single-user portfolio demo, ASVS V2/V4 waived); zod validation + length caps bound the abuse surface; documented in README (Plan 05) |
| T-01-03 | Tampering | SQL injection via name/url | mitigate | Drizzle parameterized inserts only; no sql template interpolation of user input |
| T-01-06 | Tampering | Stored XSS via competitor name/URL rendered in list | mitigate | React default escaping; no dangerouslySetInnerHTML (grep-gated); name capped at 200 chars |
| T-01-04 | Information Disclosure | DATABASE_URL(_UNPOOLED) secrets | mitigate | .env untracked (git ls-files gate); verification script never prints connection strings; errors returned to UI are generic copy, never raw DB errors |
| T-01-07 | Denial of Service | Neon free-tier CU-hours via spammed inserts | accept | Single-user demo; daily-scale usage is a tiny fraction of limits (RESEARCH.md); revisit if made public-facing beyond portfolio |
