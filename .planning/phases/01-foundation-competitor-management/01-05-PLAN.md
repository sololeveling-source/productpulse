<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public internet → Vercel app | Anyone can reach the app and invoke Server Actions (no auth by design) |
| GitHub repo → Vercel build | Repo contents become the deployed artifact; secrets must never be in the repo |
| Vercel env store → runtime | DATABASE_URL lives only in Vercel's env config |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Information Disclosure | DB connection strings in repo/build logs | mitigate | Automated git ls-files gate before push; secrets entered only in Vercel dashboard; README documents the contract instead of values |
| T-01-05 | Elevation of Privilege | Publicly reachable mutation actions on production | accept | Project decision (no auth, single-user demo); risk documented in README this plan; zod + SSRF denylist bound the surface; revisit at Phase 5 when the cron endpoint adds CRON_SECRET |
| T-01-10 | Repudiation / DoS | Strangers mutating demo data on the public URL | accept | Demo-scale accepted risk; Neon data is recoverable by re-adding competitors; no PII stored |
| T-01-SC | Tampering | Supply chain at deploy (Vercel installs from lockfile) | mitigate | pnpm-lock.yaml committed; no new packages introduced in this plan |
</threat_model>
